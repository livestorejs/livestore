## What is LiveStore?

LiveStore is an event-driven data layer. Its prime use case is building complex, client-side apps like [Linear](http://linear.app/), [Figma](https://www.figma.com/) or [Notion](https://notion.so/) that can also work offline.

You can imagine LiveStore as a next-generation version of state management libraries like Zustand, Redux or MobX that:

- _persists_ state, making it available across refreshes and browser sessions
- _distributes_ (or _syncs_) state across various clients (browsers, devices, ...)

The combination of persisted and distributed state is a giant leap for creating an **amazing user experience** (UX), while also providing a **superior developer experience** (DX) through well-designed primitives and abstractions for state management.

See this talk for more info: [Sync different: Event sourcing in local-first apps](https://www.youtube.com/watch?v=nyPl84BopKc&list=PL4isNRKAwz2MabH6AMhUz1yS3j1DqGdtT).

## The core idea: Events -> State -> UI

Due to its event-driven nature, LiveStore requires a bit of a mindset shift compared to the traditional way of building apps.

### Traditional state management uses ephemeral, in-memory state

Traditional state management works like this: you dispatch actions that update an in-memory store, and your UI reacts to changes. But that state vanishes when the user refreshes or closes the browser. Add persistence and you need to manage local storage which essentially serves as a secondary database to the data you've stored in the cloud. Add sync and you're dealing with conflict resolution, offline queues, and backend integration.

### LiveStore persists events which materialize into state

LiveStore handles all of this through one unified pattern: **event sourcing**.

```
User action → Event → Eventlog → SQLite state → UI
                         ↓
                  Sync to other clients
```

Instead of mutating state directly, you commit **events** that _describe_ what happened. These events are persisted to an **eventlog** (like a git history) and automatically materialized into a local **SQLite database** that your UI queries reactively.

:::note
While the majority of apps will probably use SQLite as the data store for persistence, LiveStore is flexible enough to materialize state into other targets as well (e.g. file systems).
:::

## A practical example

Let's walk through a simple example of a todo list with LiveStore and React:

### Define your schema

At the core of every app built with LiveStore, you have a _schema_ which consists of three parts:

- **Events**: describe what can happen in your app
- **State**: defines how data is stored in your app
- **Materializers**: determines how events are mapped to state in your app

Here's an example:

```ts
// schema.ts
import { Events, Schema, State, makeSchema } from "@livestore/livestore";

// 1. Define events (the things that can happen)
export const events = {
  todoCreated: Events.synced({
    name: "v1.TodoCreated",
    schema: Schema.Struct({
      id: Schema.String,
      text: Schema.String,
    }),
  }),
  todoCompleted: Events.synced({
    name: "v1.TodoCompleted",
    schema: Schema.Struct({ id: Schema.String }),
  }),
  todoDeleted: Events.synced({
    name: "v1.TodoDeleted",
    schema: Schema.Struct({ id: Schema.String }),
  }),
};

// 2. Define state (SQLite tables)
export const tables = {
  todos: State.SQLite.table({
    name: "todos",
    columns: {
      id: State.SQLite.text({ primaryKey: true }),
      text: State.SQLite.text({ default: "" }),
      completed: State.SQLite.boolean({ default: false }),
      deleted: State.SQLite.boolean({ default: false }),
    },
  }),
};

// 3. Define materializers (how events become state)
const materializers = State.SQLite.materializers(events, {
  "v1.TodoCreated": ({ id, text }) => tables.todos.insert({ id, text }),
  "v1.TodoCompleted": ({ id }) =>
    tables.todos.update({ completed: true }).where({ id }),
  "v1.TodoDeleted": ({ id }) =>
    tables.todos.update({ deleted: true }).where({ id }),
});

const state = State.SQLite.makeState({ tables, materializers });
export const schema = makeSchema({ events, state });
```

### Usage on the frontend

LiveStore comes with integrations for all major frontend frameworks, e.g. for [React](../../framework-integrations/react-integration) or [Vue](../../framework-integrations/vue-integration).

The `queryDb` function creates a [reactive query](../../building-with-livestore/reactivity-system) which updates automatically when its data in the database changes. Here's how to use it in React:

```tsx
// TodoApp.tsx
import { useQuery, useStore } from "@livestore/react";
import { queryDb } from "@livestore/livestore";
import { events, tables } from "./schema";

// Define a reactive query
const visibleTodos$ = queryDb(() => tables.todos.where({ deleted: false }), {
  label: "visibleTodos",
});

export function TodoApp() {
  const { store } = useStore();
  const todos = useQuery(visibleTodos$); // Reactively updates

  const addTodo = (text: string) => {
    store.commit(
      events.todoCreated({
        id: crypto.randomUUID(),
        text,
      })
    );
  };

  const completeTodo = (id: string) => {
    store.commit(events.todoCompleted({ id }));
  };

  return (
    <div>
      <button onClick={() => addTodo("New todo")}>Add</button>
      {todos.map((todo) => (
        <div key={todo.id} onClick={() => completeTodo(todo.id)}>
          {todo.completed ? "✓" : "○"} {todo.text}
        </div>
      ))}
    </div>
  );
}
```

Notice there's no loading state for queries—SQLite runs in-memory on the main thread, so reads are synchronous and instant.

## Why events?

If you've used Redux, this pattern will feel familiar:

Events are like actions, materializers are like reducers, and the SQLite state is like your store. 

But there are key differences:

| Redux                                              | LiveStore                                   |
| -------------------------------------------------- | ------------------------------------------- |
| Actions dispatch → reducers update in-memory state | Events commit → materializers update SQLite |
| State lost on refresh                              | Events persisted locally                    |
| Sync requires external setup                       | Sync built-in via eventlog                  |
| Fixed state shape                                  | Query any shape with SQL                    |

The eventlog is the core of LiveStore and gives you several advantages:

- **Persistence**: Events survive page refreshes and app restarts
- **Sync**: Events replay identically across devices
- **History**: Full audit trail of every change (enabling time travel and help with debugging)
- **Flexibility**: Change your queries without migrations—just materialize differently


## How sync works

You can enable syncing in LiveStore, which distributes your state across various other clients (these can be other browsers, apps, devices, servers—anything that can connect to your store via an _adapter_).

When you enable sync, LiveStore uses a push/pull model inspired by git:

1. Local events are committed immediately (optimistic updates by default)
1. Events sync to a central backend when online
1. Other clients pull new events and materialize them locally
1. Conflicts resolve deterministically (last-write-wins by default, or custom logic)

Here's an example that syncs your state via Cloudflare:

```ts title="/src/livestore/livestore.worker.ts" lang="ts"
import { makeWorker } from '@livestore/adapter-web/worker'
import { makeWsSync } from '@livestore/sync-cf/client'

import { schema } from './schema.ts'

makeWorker({ 
  schema,
  sync: {
    backend: makeWsSync({ url: `${location.origin}/sync` }),
  }
})
```

That's all the configuration needed—your app works offline automatically. When connectivity returns, LiveStore syncs pending events and reconciles state.

## When LiveStore fits

LiveStore works well for:

- **Productivity apps** (todo lists, note-taking, project management)
- **Collaborative tools** with offline support
- **Apps with complex local state** that need SQL-level queries
- **Cross-platform apps** (web, mobile, desktop, server)

LiveStore may not fit if:

- Your data must live on an existing server database (consider [ElectricSQL](https://electric-sql.com) or [Zero](https://zero.rocicorp.dev))
- You're building a traditional client-server app without offline needs
- You need unbounded data that won't fit in client memory

If you're unsure, go through our [evaluation exercise](./when-livestore) to find out whether LiveStore is a good fit for your project.

## Key concepts summary

| Concept          | What It Is                                                           |
| ---------------- | -------------------------------------------------------------------- |
| **Event**        | An immutable record of something that happened (e.g., `TodoCreated`) |
| **Eventlog**     | The ordered history of all events—your source of truth               |
| **Materializer** | A function that transforms an event into SQL operations              |
| **State**        | SQLite tables derived from the eventlog                              |
| **Query**        | A reactive subscription to SQLite data                               |
| **Store**        | The main interface for committing events and querying state          |
