# Ruleset reconciliation — Spec

How the repository branch ruleset is kept converged with its committed
desired-state file, using an org-owned GitHub App as the privileged identity.
Decisions and their rejected alternatives live in
[decisions/0001](./decisions/0001-github-app-definition-as-iac.md); GitHub
platform limits in
[reference/github-app-platform-constraints](./reference/github-app-platform-constraints.md).

> Status: design agreed, not yet implemented. Build scope tracked in
> [open-questions.md](./open-questions.md).

## Desired-state pipeline

```
genie (TS)  ──►  .github/repo-settings.json  ──►  GitHub App applies  ──►  live ruleset
 (authoring)        (committed desired state)      (reconcile workflow)     (GitHub)
```

The required-status-check list in `repo-settings.json` is generated by Genie from
the CI job list, so the desired state is already declarative, code-generated, and
version-controlled. The only missing IaC pillar is automated **apply**, which this
subsystem adds.

## Identity: the reconcile GitHub App

- **One org-owned App** (`livestorejs`), `administration: write` only, installed
  on `livestore` and `livestore-contrib`. Used purely to mint short-lived
  installation tokens — no webhook, no events.
- **Definition as-code:** the App is registered from a committed **App Manifest**
  (`name`, `default_permissions: { administration: write }`, `default_events:
  []`, `public: false`, no `hook_attributes`). The manifest is the reviewable
  source of truth for the App's definition.
- **Config vs secret:**
  - App ID and Client ID are **not secret** → committed as constants in the
    reconcile workflow's Genie source (reviewable, diffable).
  - The App **private key** is the single unavoidable secret → an **org-level
    Actions secret restricted to the two repos** (`LIVESTORE_RULESET_APP_KEY`).
    No external secret manager is introduced.
- **Token minting:** the workflow mints an installation token from
  `app-id + private-key` (via a `create-github-app-token`-style step), then runs
  `mono github rulesets sync` with that token.

## Reconcile control flow

A dedicated workflow **`.github/workflows/repo-settings.yml`** (genie source
`.github/workflows/repo-settings.yml.genie.ts`; single responsibility, sole
holder of the App secret). Devenv tasks: `github:rulesets:sync` (apply),
`github:rulesets:plan` (dry-run), `github:rulesets:check` (verify),
`github:app:check` (App drift). Jobs:

- **Apply on merge:** on push to `main` path-filtered to
  `.github/repo-settings.json`, mint a token and run `mono github rulesets sync`
  (apply), then re-run the drift check as a **post-apply verify**. A residual
  drift after apply is a real, alertable failure.
- **Scheduled backstop:** the same apply runs on a schedule to correct
  out-of-band console edits the merge trigger cannot observe.
- **PR plan:** on pull requests touching `repo-settings.json`, run
  `mono github rulesets sync --dry-run` as an **informational, non-gating**
  check that shows the diff that will apply on merge.
- **Gate removal:** the former `ruleset-drift-check` job is removed from
  `ci.yml`. It was never a *required status check* (it is absent from
  `requiredCIJobs`), so it never blocked merges — it wedged releases only because
  its failure set the whole `ci` run's conclusion to `failure`, and
  `publish-snapshot-version` keys off `workflow_run.conclusion == 'success'`.
  Moving reconcile+verify into the dedicated workflow removes any ruleset job
  from `ci`, so ruleset drift can no longer affect the `ci` conclusion. Live
  drift is reconciled by the merge, not policed before it.

## App-definition drift-check

Symmetric with the ruleset drift-check: a task reads the live App via `GET /app`
(App JWT auth) and diffs its `default_permissions` / `default_events` against the
committed manifest. Runs in the scheduled backstop. Because GitHub offers no API
to update an App's permissions, a detected App drift surfaces a **manual action
required** diagnostic rather than converging.

## Manual provisioning — the non-IaC surface (documented, not automated)

GitHub imposes an irreducible manual lifecycle for Apps. These steps are **the
complete list** of what is not IaC; everything else above is declarative and/or
automated. Each must be performed once by an org owner and recorded:

1. **Register the App** from the committed manifest (browser handshake:
   `POST` manifest → redirect → exchange temp `code`). Generates App ID +
   private key.
2. **Store the private key** as the org Actions secret
   `LIVESTORE_RULESET_APP_KEY`, restricted to `livestore` + `livestore-contrib`.
3. **Record the App ID / Client ID** as constants in the reconcile workflow's
   Genie source (non-secret).
4. **Install the App** on both repos with access to the ruleset (Administration)
   permission.
5. **On any later permission change**: edit `default_permissions` in the App
   settings UI (no API), which forces installations to re-consent; then update
   the committed manifest to match so the drift-check passes.

The App-definition drift-check exists so that divergence between steps 1/5 and
the committed manifest is *detected* rather than silent.

## Component ownership (shared helper, local enrollment)

Following repo-architecture R17: the manifest, workflow, tasks, and drift-checks
are authored once as shared helpers in `livestore` (core). `livestore-contrib`
enrolls by installing the same App and adding the generated workflow; each repo
owns its own `repo-settings.json`.
